# Royal Cooking — Shawarma Restaurant (HTML5 Playable)

PixiJS + TypeScript. Маленький playable (≤2–5 МБ), длительность сессии 15–30 сек.

## Технические рамки

- **Базовое разрешение: 1280×720** (landscape). Один PSD на обе ориентации — отдельной портретной композиции не будет.
- **Адаптация под landscape и portrait — cover, не contain.** Без чёрных полос: world-сцена увеличивается до полного покрытия viewport, лишние края обрезаются. В сильных аспект-несоответствиях (например, 9:16 phone) обрезается значительная часть по горизонтали — это принято.
- **UI приклеен к краям экрана и не двигается при смене ориентации.** Каждый UI-элемент имеет `UiAnchor { hAnchor, vAnchor, offsetX, offsetY, w, h }` — ставит его в нужный угол/край вьюпорта и масштабируется тем же cover-фактором, что и world.
- **Два корня в `GameScene`:**
  - `worldRoot` — фон, кухня, клиенты. Cover-scale + центрирующий offset (может уйти за границы вьюпорта).
  - `uiRoot` — coin HUD, sound button, hint, CTA. Координаты в screen-space, пересчитываются на каждом resize.
- **PSD как source of truth для world:** все позиции и размеры world-слоёв — `LayerSpec = { x, y, w, h }` из `psd/lvl.psd`, helper `applySpec(sprite, spec)` форсит `sprite.width/height` (PSD-смартобъекты non-uniformly масштабированы, ≈×0.85 ширина / ×0.90 высота).

## Геймплей

Игрок обслуживает клиентов в шаурма-ресторане.

Цикл:
1. Клиент появляется на сцене, над ним — бабл заказа.
2. В бабле одна или несколько позиций.
3. Игрок тапает по станциям/слотам ингредиентов на столе готовки.
4. Готовое блюдо появляется на тарелке.
5. Тап по готовой тарелке — она летит к подходящему клиенту.
6. Когда все позиции заказа доставлены, клиент уходит, прилетают монеты.

### Контент

- Основное блюдо — **шаурма / гирос**.
  - Обязательно: лепёшка, мясо.
  - Опционально: картошка фри, помидоры, огурцы.
- **Сок** — отдельная позиция-напиток, без сборки.
- **Мясо** жарится на вертикальном вертеле, готовые порции копятся в сковородке. Тап по сковородке кладёт порцию на активную тарелку.
- Игрок собирает только то, что есть в активных заказах.
- Матчинг тарелка↔клиент — автоматический по рецепту (forgiving, без штрафа за «не тому»).

## Архитектура

Единственная сцена, плоская структура, прямые связи через конструкторы и колбэки.
**Без ECS, FSM-библиотек, DI, scene-роутера, event bus, общих базовых классов.**

### Структура папок

```
src/
  index.ts          // бутстрап, монтирование canvas
  App.ts            // PIXI.Application, resize, держит GameScene
  config.ts         // все константы: тайминги, цены, патиенс, задержка хинта
  assets.ts         // манифест + Assets.load + типизированные геттеры
  layout.ts         // computeLayout(w,h) → координаты якорей (portrait/landscape)
  recipes.ts        // словарь рецептов (данные)
  redirect.ts       // обёртка над SDK сетки (IronSource/Mintegral/Vungle/...)

  scene/
    GameScene.ts    // оркестратор: ticker, связывает все подсистемы

  game/
    Client.ts       // вход, бабл заказа, патиенс, уход
    ClientQueue.ts  // спавн и очередь
    Order.ts        // массив OrderItem { recipe, delivered }
    Kitchen.ts      // станции, активная тарелка, очередь готовых тарелок
    MeatSpit.ts     // таймер прожарки, счётчик порций в сковородке
    Plate.ts        // одна тарелка, тап → лететь к клиенту

  ui/
    Hint.ts         // pointAt(target) — рука-подсказка
    CoinsHud.ts     // счётчик + анимация прилёта монет
    CTA.ts          // энд-кард + кнопка → redirect.openStore()
```

~14 файлов.

### Ответственности

- **App** — поднимает Pixi, ресайз, visibilitychange, держит `GameScene`.
- **GameScene** — единственная сцена. Создаёт `ClientQueue`, `Kitchen`, `Hint`, `CoinsHud`, `CTA`. Подписана на `app.ticker`. Хранит шаги туториала (линейный массив индексов, не FSM).
- **config.ts** — единое место для всех чисел. Геймдизайн крутит здесь.
- **assets.ts** — один манифест, один `Assets.load`.
- **layout.ts** — чистая функция, возвращает позиции/скейлы для текущего вьюпорта.
- **recipes.ts** — литерал со словарём рецептов: `{ shawarma: { required: [...], optional: [...] }, juice: {...} }`.
- **redirect.ts** — единственная точка кастомизации под рекламную сеть. Экспорт `openStore()`.
- **Client / ClientQueue** — состояния клиента enum (`Walking | Ordering | Waiting | Served | Leaving`).
- **Order** — массив `OrderItem`, метод проверки «всё ли доставлено».
- **Kitchen** — оркестратор кухни: держит ингредиент-кнопки, `MeatSpit`, зону сока, активную тарелку, очередь готовых.
- **MeatSpit** — пассивная подсистема с `update(dt)`. Копит порции до лимита.
- **Plate** — собранное блюдо. По тапу спрашивает `ClientQueue` о подходящем клиенте, твином летит к нему, помечает `OrderItem.delivered = true`.
- **Hint** — `pointAt(target | null)`. Цели ставит `GameScene` по шагам туториала.
- **CoinsHud** — счётчик + поп-анимация.
- **CTA** — оверлей с кнопкой, вызывает `redirect.openStore()`.

### Поток `update(dt)`

`GameScene` подписывается на `app.ticker.add` и зовёт `update(dt)` на:
- `MeatSpit` — таймер прожарки;
- `ClientQueue` — таймер патиенса клиентов, спавн;
- `Hint` — таймер бездействия.

Остальное реактивное (тапы, твины) — без апдейтов.

### Шаги туториала (Hint)

Линейный список целей в `GameScene`:
1. лепёшка,
2. сковородка с мясом,
3. готовая тарелка (подача).

Дальше — следующий заказ без подсказки или хинт всплывает только при простое. После N заказов → `CTA`.

## Принципы, которым следуем

- Прямые связи лучше абстракций. 10 модулей читаются проще DI-контейнера.
- Конкретные модули вместо `BaseStation`/`Item`/`Entity`.
- Состояния — `enum` и `switch`, не FSM-библиотека.
- Все числа в `config.ts` — итерации QA сетки идут быстро.
- Один `redirect.ts` изолирует различия сеток, остальной код ничего о них не знает.
- Forgiving-логика: автоматчинг, без штрафов, без «проиграл».
